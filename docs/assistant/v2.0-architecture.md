## Preamble

The previous document described a voice search feature. This document describes a product.

The distinction is not semantic. A voice search feature replaces the search bar. A leasing
assistant replaces the leasing agent. These are different in scope, in technology, in business
impact, and in what it takes to get right. This document is written with that scale in mind.

Before any technology decision, one product truth must be established: **the assistant's job
is to get a renter into the right home.** Not to demonstrate AI capability. Not to be
impressive. Not to have a personality. Its one measure of success is whether renters who use
it sign leases faster, with higher satisfaction, than renters who do not. Every architectural
decision in this document flows from that constraint.

## 1. The Ideal Conversational Experience

### 1.1 What the User Should Feel

The user opens Emlakie and sees a small, permanent "Talk to a leasing assistant" button. They
press it. A voice starts speaking — not a robotic voice, a calm, knowledgeable human-sounding
voice — and says something specific:

*"Hi, I'm here to help you find your next home. What area are you looking in?"*

The user answers naturally. The voice listens. When the user finishes, it responds within half
a second. The conversation moves like a conversation, not like a search form read aloud.

The assistant asks exactly the questions a skilled leasing agent would ask and in the same
order. It does not ask for everything at once. It listens for cues — if the user mentions a
dog, it adds pet-friendly to the list without being asked.

When it has enough to work with, it says: *"I found 12 places in Bakersfield that fit what
you've described — there are two that I think are particularly worth your time. Want me to
tell you about them?"* Then it talks about the listings the way a good agent would: what makes
them notable, not just their data.

### 1.2 What the Assistant Is Not

It is not a customer service chatbot. It is not an encyclopedia. It is not always-on
listening. It is a professional you call to help you find a home.

### 1.3 The Conversational Flow Architecture

Five natural phases:

1. **Orientation (0–30s)** — Location, timeline, property type
2. **Refinement (30–90s)** — Priority probing based on candidate listings
3. **Presentation (60–180s)** — 2–3 specific listings with reasons
4. **Evaluation (variable)** — Objection handling, pivots, follow-up questions
5. **Action (variable)** — Tour scheduling, saving, landlord contact

## 2. Technology Stack

### 2.1 Recommendation: Hybrid Architecture (OpenAI Realtime + Claude)

- **OpenAI Realtime API** — voice layer (STT, VAD, TTS, interruption, real-time turn-taking)
- **Claude Sonnet 4.6** — reasoning layer (filter normalization, recommendation explanation,
  personalization context, session summarization)

### 2.2 Technology Layer Map

| Layer | Technology |
|---|---|
| Voice Activity Detection | OpenAI Realtime (built-in) |
| Speech-to-Text | OpenAI Realtime (Whisper-based) |
| Conversational LLM | GPT-4o Realtime |
| Reasoning / Recommendations | Claude Sonnet 4.6 |
| Text-to-Speech | OpenAI Realtime (built-in) |
| Search Engine | Existing Supabase pipeline (unchanged) |
| Session State | Server-side Redis (Upstash) |
| Long-term Memory | Supabase (renter_profiles table) |
| WebSocket Transport | Persistent server (Fly.io → Vercel WS) |
| Analytics | Existing GA4 |

### 2.3 Latency Target

Sub-500ms from user finishing speech to assistant beginning response (P50).
ChatGPT Voice achieves ~320ms. Phase 1 target: <500ms.

## 3. Browser Responsibilities

- Capture microphone audio (Web Audio API, 16kHz mono PCM)
- Stream audio frames over WebSocket to server
- Receive audio chunks and play via AudioContext
- Manage visual assistant state (listening / thinking / speaking / idle)
- Render listings panel on assistant's signal
- Handle session end

**Browser does NOT do:** STT, LLM inference, session state, search logic, personalization.

---

## 4. Server Responsibilities

**Session Manager (Node.js WebSocket server):**
- Accept browser WebSocket connections
- Maintain OpenAI Realtime WebSocket session
- Bridge audio bidirectionally
- Manage session lifecycle
- Enforce session timeouts

**Context Engine (Claude Sonnet 4.6):**
- Generate system prompt at session start (enriched with user profile)
- Inject enriched context at key conversation decision points
- Generate session summary for memory store after session end

**Function Executor (server-side tools):**
- `search_listings(filters)` — existing Supabase query
- `get_listing_details(listing_id)`
- `explain_recommendation(listing_ids[], context)` — Claude reasoning
- `save_listing(listing_id)`
- `start_tour_request(listing_id)`
- `update_session_preferences(preferences)`
- `recall_user_profile()`

## 5. Conversation Memory

### 5.1 In-Session (Ephemeral — Redis, 4h TTL)

Structured SessionState: detected language, filter state, presented/rejected/saved listing IDs,
conversation phase, behavioral signals. **Never stores transcript text.**

### 5.2 Cross-Session (Persistent — Supabase renter_profiles)

Built from structured session state (not transcripts) after each session ends. Contains stable
preferences, inferred preferences, and behavioral patterns.

---

## 6. Personalization

Three levels: explicit stated preferences → inferred soft preferences → behavioral signals.
Injected as system prompt enrichment at session start. Never uses language as a proxy for
protected class. Fair Housing compliance is a design constraint at every layer.

---

## 7. Integration With Emlakie Search

The assistant calls `search_listings()` which runs the existing `getListings()` from
`lib/api.ts` unchanged. Claude adds soft-ranking on top of results and generates explanation
text. The assistant terminates at URL/listing-ID output — the existing search pipeline
handles all rendering.

---

## 8. System Learning

- Per-session: assistant adapts within conversation
- Per-user: renter_profiles updated after every session
- Aggregate: behavioral signals (listing presented → saved → tour requested) inform
  recommendation ranking over time. No transcript content ever used.

---

## 9. Telemetry

GA4 behavioral events only. Never logs transcript text, language tied to user identity,
audio, or protected class proxies. Session telemetry uses anonymous session IDs.
