## The Case for Multilingual as Foundation

Bakersfield's population is 52% Hispanic or Latino. The LA metro — Emlakie's natural expansion
market — is among the most linguistically diverse urban regions on earth. Persian is the fourth
most spoken language in LA County. Armenian communities are concentrated in Glendale, Pasadena,
and Burbank. Russian-speaking communities span the San Fernando Valley. Arabic-speaking
communities exist across multiple LA cities.

A leasing assistant that speaks a renter's language is not an accessibility feature. It is a
product moat. Zillow has a search bar. It does not speak Persian.

Multilingual support affects data normalization, memory architecture, Fair Housing guardrails,
and system prompt design. It cannot be retrofitted. It must be designed in from the beginning.

## 2.6 OpenAI Realtime API: Multilingual Capability Assessment

GPT-4o (underlying OpenAI Realtime) is a genuinely multilingual model — not a translation
overlay. It detects language from the first 3–5 words, responds in the detected language,
holds full conversations in non-English, and handles code-switching naturally.

### Language-by-Language Assessment

| Language | STT Quality | LLM Quality | TTS Quality | Phase 1 Ready? |
|---|---|---|---|---|
| English | Excellent | Excellent | Excellent | Yes |
| Spanish | Excellent | Excellent | Excellent | Yes |
| Russian | Excellent | Excellent | Very Good | Yes |
| Arabic | Very Good | Very Good | Good | Yes |
| Persian/Farsi | Good | Good | Good | Yes, with monitoring |
| Armenian | Functional | Functional | Fair | Yes, with Beta label |

Persian note: Modern Standard Persian is well-supported. Iranian Farsi with colloquialisms
is weaker. Active quality monitoring required from Day 1.

Armenian note: Eastern Armenian > Western Armenian support. Launch with Beta label and
lower confidence thresholds (assistant asks more clarifying questions).

### Claude's Multilingual Responsibilities

1. Filter normalization (any language → English database values)
2. Recommendation explanation generation (in user's language, proper nouns anchored)
3. Session context enrichment (system prompt always in English)
4. Proper noun anchoring (addresses, city names never translated)

## 12. Multilingual Architecture

### 12.1 Core Principle

**User's language = interaction language. English = data layer language.**
These concerns never bleed into each other. The normalization bridge is the only connection.

### 12.2 Architecture Stack

```
User speaks → OpenAI Realtime STT (detects language) →
GPT-4o (maintains conversation in source language) →
Function call: search_listings(raw_utterance, session_state) →
Normalization Bridge (Claude): any language → English filter values →
Supabase query (always English) → English listing data →
Recommendation Engine (Claude): generates reasoning in USER'S LANGUAGE,
proper nouns anchored as untranslatable →
OpenAI Realtime TTS → User hears response in their language
Visual listing cards always render in English (as stored in DB)
```

### 12.3 Language Detection

Implicit — no language selector shown to user. GPT-4o detects from first utterance.
Session state records `detected_language` (ISO 639-1 code). Language locked after 2+
exchanges. Single-language borrowings (English words in Persian sentence) are not switches.

### 12.4 Normalization Bridge Rules

**Property types → closed enum values (always English):**
- "apartamento" / "آپارتمان" / "шqqa" / "квартира" / "բնakaran" → `apartment`
- "casa" / "خانه" / "дом" → `house`
- "estudio" / "استودیو" → `studio`

**Prices → numeric extraction** (all language variants, validated $300–$20,000/month)

**Bedroom counts → cultural mapping** (Russian "двухкомнатная" ≠ "two-bedroom" in US convention)

**Proper nouns → NEVER translated.** Cities matched to canonical DB names via lookup table.
Phonetically closest match returned with low-confidence flag if no exact match.

**Amenities → closed list** (same English strings as stored in DB, mapped from any language)

### 12.5 Recommendation Text Rules

1. Proper nouns wrapped as untranslatable, inserted verbatim
2. Addresses spoken as reference data, never localized
3. Prices stated numerically ($ always, no currency conversion)
4. Listing facts never paraphrased beyond translation of the term itself
5. Legal / immigration interpretations never added regardless of user's language

### 12.6 Visual UI

Listing cards: always English (as stored). Legal documents: always original language.
RTL rendering: activated for Arabic and Persian when detected. `dir="rtl"` on assistant
panel only — listing cards and main UI remain LTR.

### 12.7 Mixed-Language (Code-Switching)

Common in immigrant communities. Handled at GPT-4o token level. System prompt instructs:
match dominant language, not the language of the last word spoken. Single borrowings are
not language switches. Session language locks after 2+ confirmed exchanges.

## Updated Phased Rollout

### Phase 1 — Foundation + English + Spanish (8–10 weeks)
- OpenAI Realtime with no hard language restriction
- Normalization bridge: English + Spanish vocabulary
- City lookup: Spanish-language variants for all Emlakie markets
- RTL support implemented (inactive until Phase 2)
- Fair Housing multilingual guardrails in system prompt (legal review before launch)
- Language code logged per session (never tied to user identity)
- Spanish TTS quality verified by native speaker before launch
- Full test coverage: English + Spanish

### Phase 2 — Persian/Farsi + Russian (4–6 weeks post-Phase 1)
- Normalization bridge: + Persian + Russian vocabulary
- RTL rendering activated for Persian
- Russian bedroom count convention mapping
- Persian + Russian TTS native-speaker evaluation
- Beta label for Persian sessions
- 5% session sampling for quality monitoring

### Phase 3 — Armenian + Arabic (4–6 weeks post-Phase 2)
- Normalization bridge: + Armenian + Arabic
- Arabic RTL (same rendering path as Persian)
- Eastern vs. Western Armenian dialect handling in system prompt
- LA-area Arabic community city patterns in lookup table
- Armenian Beta label at launch
- Community partner testers recruited for Armenian and Arabic live beta

### Phase 4 — Language Intelligence (ongoing)
- User-selectable language preference (opt-in account setting)
- Language-specific vocabulary improvements from monitoring
- Cascade pipeline evaluation for Armenian (weakest OpenAI STT)
- Next candidates: Vietnamese, Korean, Tagalog, Mandarin (LA expansion)
- Language quality scorecards: monthly internal, quarterly legal/product review

## Multilingual Latency Risks

| Risk | Impact | Mitigation |
|---|---|---|
| Normalization bridge adds latency | +200–400ms on function calls | Claude Haiku for normalization; Sonnet for recommendations |
| Lower STT accuracy for Armenian | User must repeat → full turn cycle | Lower confidence threshold; more clarifying questions |
| Unnatural TTS for Persian/Armenian | User trust degradation | Beta labeling; monitor abandonment by language |
| City name resolution failure | No results | Pre-loaded transliteration lookup table; phonetic fallback |
| Code-switch misdetection → wrong filter | Incorrect Supabase query | Low-confidence fields → null (no filter) rather than guess |
| Proper noun corruption in TTS | User misidentifies property | Visual card always simultaneous; user sees correct text |

Revised function call latency budget (multilingual):
- Supabase query: 150ms | Normalization: 250ms | Recommendation generation: 500ms
- Serialization: 50ms | **Total: 950ms** (within acceptable threshold)

## Multilingual Testing Requirements

### Per-language before launch:
- All 6 property types in target language → correct enum
- All 15 amenities in target language → correct enum
- Price ranges in colloquial forms → correct numeric extraction
- Bedroom counts with cultural conventions → correct mapping
- Common code-switching patterns → correct dominant-language detection
- City names in native transliteration → correct canonical DB name
- Complete orientation → presentation → action flow in target language
- Proper noun pass-through (addresses unchanged)
- Immigration question → correct decline response in target language
- Protected class language → not reflected in recommendations

### Audio quality (human evaluation by native speaker):
- TTS naturalness score (1–5)
- TTS intelligibility (every word understood?)
- English address pronunciation in target-language voice

### Regression:
Every system prompt or normalization change must be regression-tested against frozen
conversation sets in all active languages before deployment.

### Ongoing monitoring:
5% of sessions sampled, reviewed by fluent speaker. Sessions where user repeated same
information 3+ times flagged as potential comprehension failure.

## Multilingual Fair Housing Guardrails

### Language Is Not a Proxy for Protected Class

Language must never — directly or indirectly — be used to infer national origin, race,
religion, immigration status, familial status, or any other protected characteristic.

**Explicit prohibitions:**
- Same listings, same order, same ranking for identical stated preferences regardless of
  language spoken
- No inference of household composition, citizenship, income source, or visa status from language
- No neighborhood steering based on language
- Arabic sessions: no religion inference; Persian sessions: no national origin inference

### Required System Prompt Language (legal review required before launch)

"You are assisting renters with property searches. You must not make any assumptions about a
user's national origin, race, religion, citizenship status, immigration status, or familial
status based on the language they speak or any other characteristic. You must not refer to
these characteristics in any recommendation, question, or response. If a user asks about
documentation requirements, visa status, or eligibility criteria, respond only with: 'That's
a question best asked directly to the landlord — I can help you contact them.'"

### Telemetry

`detected_language` code logged per anonymous session for quality monitoring only.
Not stored in renter_profiles. Not used for targeting, segmentation, or recommendations.
One exception: explicit user-set language preference (opt-in, Phase 4+).

### Proper Noun Accuracy

Misidentified address or city = potential material misrepresentation. Proper noun anchoring
rules in Section 12.5 are legal + trust requirements, not quality preferences. All proper
nouns verified against canonical DB values before any recommendation is spoken.

## The Competitive Moat

Zillow, Apartments.com, Trulia: English-first, translation as afterthought.
Their translation features are machine-translation overlays on English interfaces.
They do not understand what a renter is asking for in Persian.
They do not know that "двухкомнатная" describes a different apartment than "two-bedroom."

A renter navigating a lease in their second language will use the tool that understands them.
The assistant that speaks Persian is not competing with a Persian Zillow.
It is competing with a relative or friend who used to translate. It can be better.
